# 每晚讀書會（book-club）規格 v1 — 2026-07-04

> Andy 需求原話：每天晚上收到一本書的內容介紹，像參與一場讀書會；晚上 7 點透過 TG 傳一個網頁統整內容；主題＝自我成長、職場成長、人際關係、邏輯思考；**要詳細，不是過過水的內容**。已拍板方案 1（內容庫＋輕排程）。

## 產品形態

- 靜態網站（純 HTML，無框架），手機優先的長文閱讀體驗；GitHub Pages 部署。
- 每天一本書一頁：`/books/<slug>.html`；`/` 是最新一篇＋歸檔列表。
- `manifest.json`：`[{date, slug, title, author, theme, hook, url}]`——排程器只讀這個檔。
- 視覺方向待 Andy 選：1️⃣ 書房夜讀風（暖米紙＋深咖啡＋Noto Serif TC＋墨綠黑夜模式）／2️⃣ 極簡編輯部風。選定後照 ui-design-flow 關卡走（MASTER.md 落檔）。

## 內容規格（深讀版，每本 3000~4500 字繁中，10~15 分鐘）

固定結構（content/books/*.md frontmatter + 章節）：
1. **作者與時代背景**——這個人憑什麼寫這本書、書誕生的脈絡
2. **這本書回答什麼問題**——一段話說清楚，讀者痛點對號入座
3. **核心概念深講 ×3~5**——每個概念：主張／書中原始案例或研究／為什麼成立（機制）／反例或限制
4. **概念之間怎麼串**——整本書的邏輯骨架，不是零散清單
5. **實作指南**——明天就能做的具體步驟（場景化：職場/生活各至少一例）
6. **常見誤讀**——這本書最常被斷章取義的地方
7. **延伸對照**——與同主題 2~3 本書的異同（帶到未來書單）
8. **金句 ×3**（附原文若為英文書）
9. **討論題 ×2**——開放式，Andy 回 TG 訊息即開聊（讀書會互動核心）

品質紅線：內容須忠於原書實際論點與案例，不虛構書中不存在的研究/數字；引用研究時標明出處於書中章節；用 Andy 的語言（白話＋比喻，避免書腔翻譯味）。

## 書單策展（四主題輪替）

首批 7 本（第一週）：原子習慣（自我成長）→ 深度工作力（職場）→ 被討厭的勇氣（人際）→ 快思慢想（邏輯）→ 與成功有約（自我成長）→ 卡內基人性的弱點（人際）→ 原則 Principles（職場/思考）。
補書節奏：每週補 7 本；Andy 隨時插隊指定書。備選池（30+）記在 content/booklist.md。

## 產製流程（品質把關）

1. Fable 親寫 1 本範本（原子習慣）定調深度與語氣。
2. 其餘每本派 **opus** subagent 照範本結構撰寫（書籍內容屬品味＋準確度敏感，不用 sonnet）；prompt 附範本全文＋品質紅線。
3. Fable 抽驗每批：結構齊全、無虛構、語氣一致；抽驗不過該本重寫。
4. build.mjs：markdown → HTML（沿用 parseArticle 式輕量解析或直接寫渲染器）＋ manifest.json 更新。

## 排程（接現有 Apps Script「股票排程」專案）

- `dispatchBookClub()`：每日 19:00 Asia/Taipei 觸發——fetch site `manifest.json`，找今天日期的條目，發 TG：📚 書名＋作者＋一句鉤子＋網頁連結；找不到當日條目→發提醒給 Andy（庫存空了）。
- 部署方式＝GitHub Pages（gh-pages 分支，npm run deploy 慣例）；**不用 GitHub Actions cron**（2026-07-04 罷工前科）。
- Andy 動作（一次性）：把 dispatchBookClub 加進現有 Code.gs＋跑 createTriggers（與待辦中的股票排程更新可一次做完）。
